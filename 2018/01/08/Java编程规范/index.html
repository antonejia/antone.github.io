Java编程规范

Veröffentlicht am 2018-01-08

一. 编程规约

  1. 关于基本数据类型与包装数据类型的使用标准如下:

    • 【强制】所有的POJO类属性必须使用包装数据类型。
    • 【强制】RPC方法的返回值和参数必须使用包装数据类型。
    • 【推荐】所有的局部变量使用基本数据类型。

    说明:POJO 类属性没有初值是提醒使用者在需要使用时,必须自己显式地进行赋值,任何 NPE 问题, 或者入库检查,都由使用者来保证。

    正例: 数据库的查询结果可能是 null,因为自动拆箱,用基本数据类型接收有 NPE 风险。

    反例:比如显示成交总额涨跌情况,即正负 x%,x 为基本数据类型,调用的 RPC 服务,调用不成功时,返回的是默认值,页面显示为 0%,这是不合理的,应该显示成中划线。所以包装数据类型的 null 值,能 够表示额外的信息,如:远程调用失败,异常退出。

    解读:根本原因在于基本数据类型必须有值,而基本数据类型的包装类可以是 null。使用基本数据类型的 包装类代替基本类型,可以避免出现 null 值时因无法放入基本数据类型而抛出的 NullPointerException。对于不能停机升级的互联网业务尤为重要,因为数据库表在线升级一般采用增加 可空字段的手段,老数据的新字段一般是 null,所以数据对象(DO)最好使用基本数据类型的包装类,由 应用层逻辑来保证数据完整性。

  2. 单行字符数限制不超过 120 个,超出需要换行,换行时遵循如下原则:

    • 方法调用时,多个参数,需要换行时,在逗号后进行。
    • 在括号前不要换行,见反例。

  4. 【强制】关于 hashCode 和 equals 的处理,遵循如下规则:

    • 只要重写equals,就必须重写hashCode。
    • 因为Set存储的是不重复的对象,依据hashCode和equals进行判断,所以Set存储的对象必须重 写这两个方法。
    • 如果自定义对象作为Map的键,那么必须重写hashCode和equals。

    说明:String 重写了 hashCode 和 equals 方法,所以我们可以非常愉快地使用 String 对象作为 key 来使用。

  5. 【强制】 ArrayList 的 subList 结果不可强转成 ArrayList,否则会抛出 ClassCastException 异 常,即 java.util.RandomAccessSubList cannot be cast to java.util.ArrayList。
    说明:subList 返回的是 ArrayList 的内部类 SubList,并不是 ArrayList ,而是 ArrayList 的 一个视图,对于 SubList 子列表的所有操作最终会反映到原列表上。

  6. 【强制】使用集合转数组的方法,必须使用集合的 toArray(T[] array),传入的是类型完全一样的数 组,大小就是 list.size()。

  7. 【强制】使用工具类 Arrays.asList()把数组转换成集合时,不能使用其修改集合相关的方法,它的 add/remove/clear 方法会抛出 UnsupportedOperationException 异常。

    说明:asList 的返回对象是一个 Arrays 内部类,并没有实现集合的修改方法。Arrays.asList 体现 的是适配器模式,只是转换接口,后台的数据仍是数组。

    • 第一种情况:list.add(“yangguanbao”); 运行时异常。
    • 第二种情况:str[0] = “gujin”; 那么 list.get(0)也会随之修改。

    怎么解？ 将Arrays.asList()得到的对象重新创建新的对象。List list = Arrays.asList(str);
    ArrayList copiedList = new ArrayList<>(list);

  9. 【强制】velocity 调用 POJO 类的属性时,建议直接使用属性名取值即可,模板引擎会自动按规范调用 POJO 的 getXxx(),如果是 boolean 基本数据类型变量(boolean 命名不需要加 is 前缀),会自动调 用 isXxx()方法。

    解读:velocity 调用 POJO 类的属性时处理规则,注意布尔型基本数据类型和包装类对象之间的区别。

  10. 【推荐】单表行数超过500万行或者单表容量超过2GB,才推荐进行分库分表。

    说明:如果预计三年后的数据量根本达不到这个级别,请不要在创建表时就分库分表。

    解读:分库分表的优势:可以将压力分散到多张表和库满足互联网高并发读的需求(适合业务数据实时性 要求不会太高,跟钱有关的核心数据一定要严格考虑延时带来的后果),安全性更高,某个功能出现故障 不影响整体业务。依靠普通的 PC SERVER 架构堆机器,屌丝流逆袭小型机上的 Oracle。

    分库分表的缺点:许多聚集操作变得非常困难(group by,max,min,count),应用需要额外的程序来控 制数据路由,要仔细考虑跨库事务控制(JTA 慢,最大努力一段式提交要考虑失败手工回滚策略),基于 全局数据视图的各种系统开发都会更复杂,另外在数据迁移和升级过程中也加大了难度。所以除非数据量 很大,否则没有必要分库分表。

  11. 【强制】超过三个表禁止 join。需要 join 的字段,数据类型必须绝对一致;多表关联查询时,保证被关 联的字段需要有索引。

    说明:即使双表 join 也要注意表索引、SQL 性能。

    解读:数据类型为何要一致?表连接是十分消耗性能的动作。数据类型的一致能大量减少数据类型转换带 来的性能消耗。举例:两表日期作为连接关键字,有的同学喜欢用字符串,有的喜欢用日期类型。在做表 连接时,将带来巨大的性能消耗,其次索引问题也将浮现,索引对于数据类型转换是十分敏感的,曾经就 有由于字符串没加单引号导致没走索引继而导致锁争用问题,load 飙升。 被关联的字段为何必须要有索引?Mysql 的连接只有 NEST LOOP 一种,类似于开发同学常见的嵌套循环, 两个 FOR 循环这种。外 FOR 循环,在 Mysql 里叫驱动表;内 FOR 循环,在 Mysql 里叫被驱动表。在连接时 候 Mysql 把外循环(驱动表)中每一个值代入到内循环(被驱动表中遍历),即内循环要被检索 N 次,N 为 驱动表的行数(可能会有些 where 条件过滤一部分数据)。如果内循环(被驱动表)没有索引,也就是说 要全表扫描 N 次。

  12. 【强制】页面搜索严禁左模糊或者全模糊,如果需要请走搜索引擎来解决。

    说明:索引文件具有 B-Tree 的最左前缀匹配特性,如果左边的值未确定,那么无法使用此索引。 解读:like ‘139%’这种可能走索引,like ‘%1234%’和 like ‘%2345’这种一定不走索引。

  13. 【推荐】如果有 order by 的场景,请注意利用索引的有序性。order by 最后的字段是组合索引的一部 分,并且放在索引组合顺序的最后,避免出现 file_sort 的情况,影响查询性能。
    正例:where a=? and b=? order by c; 索引:a_b_c 反例:索引中有范围查找,那么索引有序性无法利用,如:WHERE a>10 ORDER BY b; 索引 a_b 无法 排序。

    解读:假设索引列顺序为(城市,年龄,收入),索引类似下图的逻辑
    北京 上海 广州
    / \ / \ / \
    20 25 20 25 20 25
    | | | | | |
    收入 收入 收入 收入 收入 收入
    因为索引是有序的,所以收入应该是排序好的。当提取这张表中北京人的收入从小到大结果时,这个索 引就无法被利用,为什么呢?因为他是按照北京-20 岁和北京-25 岁这个条件分别排序的。并没有按照北 京人这个单独条件来排序,自然也不可能利用到索引。解决办法是重建一个索引(城市,收入),这样 就可以利用索引来检索,各个城市收入的排序结果了。同理,范围查找也是一样,由于前面的列范围被 模糊了,而索引只能得到特定条件下的排序结果,所以无法使用索引。

  14. 【推荐】利用延迟关联或者子查询优化超多分页场景。
    说明:MySQL 并不是跳过 offset 行,而是取 offset+N 行,然后返回放弃前 offset 行,返回 N 行,那 当 offset 特别大的时候,效率就非常的低下,要么控制返回的总页数,要么对超过特定阈值的页数进行 SQL 改写。

    正例:先快速定位需要获取的 id 段,然后再关联:
    SELECT a.* FROM 表 1 a, (select id from 表 1 where 条件 LIMIT 100000,20 ) b where a.id=b.id

    解读:正例的思路是利用表的覆盖索引(参考第 6 条)来加速分页的 ID 列表查询,然后再进行表关联查询出结果集。

  15. 【强制】sql.xml配置参数使用:#{},#param#不要使用${}此种方式容易出现SQL注入。

    解读:在 iBatis 或 MyBatis 里面,“#”占位符底层会使用 PreparedStatement 来实现,“$”占位符只 是在 SQL 上进行字符串替换。而字符串拼接 SQL 是 SQL 注入的问题根源所在。
